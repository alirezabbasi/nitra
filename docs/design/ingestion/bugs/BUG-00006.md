# BUG-00006: Startup 90-Day Backfill Stalls on `failed_no_source_data`

## Status

- Resolved in code
- Runtime validation in progress

## Description

The deterministic backfill pipeline (`gap-detection` -> `backfill-worker` -> `replay-controller`) queues jobs correctly, but most jobs fail because replay currently rebuilds only from existing `raw_tick` rows and does not fetch missing broker history.

## Impact

- Required 90-day `1m` continuity often remains incomplete.
- Gap lifecycle remains stuck with many `backfill_queued` rows.
- System behavior diverges from `DEV-00013` intent ("missing-only broker backfill to completion or explicit surfaced error").

## Reproduction

1. Start stack with sparse recent `raw_tick` history (typical fresh/dev environment).
2. Let startup coverage scan create gaps and backfill jobs.
3. Query `backfill_jobs`, `replay_audit`, and `gap_log`.
4. Observe dominant `failed_no_source_data` / `failed` states and unresolved queued gaps.

## Root Cause Analysis

- `services/replay-controller/src/main.rs` aggregates bars from `raw_tick` only.
- If ticks are missing for requested ranges, job status becomes:
  - `backfill_jobs.status = failed_no_source_data`
  - `replay_audit.status = failed` with error `"no source ticks found for requested range"`.
- Broker-history fetch is not yet implemented in replay-controller path.

## Resolution Implemented

- Added venue-history adapters in `services/replay-controller/src/main.rs` for replay ranges that remain incomplete after `raw_tick` rebuild:
  - OANDA candles adapter (`/v3/instruments/{instrument}/candles`)
  - Coinbase Exchange candles + Advanced Trade public fallback
  - Capital authenticated prices adapter with session refresh
- Replay now attempts venue-history fetch before assigning terminal `failed_no_source_data` and re-checks completeness after adapter upserts.
- Added compose/env contract for replay history adapter credentials/URLs and timeout controls.
- Added persistent backfill recovery in `services/backfill-worker/src/main.rs`:
  - periodic re-enqueue of stale/orphaned `queued` rows into `replay.commands`,
  - stale `running` timeout reset back to `queued`,
  - enqueue tracking columns in `backfill_jobs` (`enqueue_count`, `last_enqueued_at`).
- Hardened replay history adapters in `services/replay-controller/src/main.rs`:
  - Coinbase Exchange/public requests now include explicit `User-Agent`,
  - OANDA instrument resolution now supports USD-suffix fallback (for example `US30_USD`, `NAS100_USD`) plus optional explicit map (`REPLAY_OANDA_INSTRUMENT_MAP`).
- Added deterministic queue-drain safeguards:
  - `backfill-worker` recovery now re-enqueues only truly stale queued jobs (stale queue/audit timestamps), not every queued row each cycle.
  - recovery selection order is deterministic oldest-first (`COALESCE(last_enqueued_at, created_at)`), preserving fair backlog drain.
  - replay-controller now supports safe parallel consumption via `REPLAY_WORKER_COUNT` (multiple consumers in the same group, partition-safe scaling).
