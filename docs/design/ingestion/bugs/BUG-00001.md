# BUG-00001: TimescaleDB Bootstrap Stops Before `raw_tick` Creation

## Status

- Resolved in code
- Runtime recovery applied for current environment

## Description

After running `docker compose down -v` and then `docker compose up -d --build`, critical TimescaleDB tables were missing, including:

- `raw_tick`
- `processed_message_ledger`

This broke chart/tick queries and ingest dedup flows.

## Impact

- `query-api` hot tick endpoint fails (`relation "raw_tick" does not exist`).
- Services relying on processed-message dedup fail (`relation "processed_message_ledger" does not exist`).
- Platform startup appears healthy at container level while core data paths are broken.

## Reproduction

1. Run `docker compose down -v`.
2. Run `docker compose up -d --build`.
3. Inspect `timescaledb` logs.
4. Observe init failure in `006_oms_lifecycle.sql` and missing relations from later scripts.

## Root Cause Analysis

TimescaleDB init executes SQL files in lexical order from `infra/timescaledb/init/`.

In `006_oms_lifecycle.sql`, bootstrap attempted:

- `SELECT create_hypertable('fills_recent', 'fill_ts', ...)`

But `fills_recent` schema had unique constraints that do not include `fill_ts`:

- `PRIMARY KEY(fill_id)`
- `UNIQUE(venue_fill_id)`

TimescaleDB rejects hypertable conversion in this case, emitting:

- `cannot create a unique index without the column "fill_ts" (used in partitioning)`

Because init scripts run with `ON_ERROR_STOP`, execution stopped at script `006`, so scripts `007` and `008` never ran, leaving `processed_message_ledger` and `raw_tick` absent.

## Resolution Implemented

Code fix:

- File: `infra/timescaledb/init/006_oms_lifecycle.sql`
- Change: removed hypertable conversion for `fills_recent` and added explicit rationale comment.

Result:

- Init no longer aborts at script `006`.
- Later scripts run normally and create:
  - `processed_message_ledger`
  - `raw_tick`
  - other market event tables.

Runtime recovery for already-booted broken environment:

- Re-executed init scripts `006`, `007`, and `008` via `psql` inside the running `timescaledb` container.

## Verification Evidence

- TimescaleDB log inspection confirmed the original failure point in `006_oms_lifecycle.sql`.
- After applying fix/recovery scripts:
  - `processed_message_ledger` exists.
  - `raw_tick` exists.
- Core query path no longer errors on missing table relations.

## Notes

- This bug is specifically exposed by fresh volume initialization (`down -v` + `up`).
- Existing persistent volumes created before this incident may hide the issue until next full rebootstrap.
