# BUG-00002: `market-normalization` Crash Loop on Duplicate Ledger Offset

## Status

- Resolved in code
- Runtime restart required

## Description

`market-normalization` repeatedly crashed during steady-state ingestion with:

- `psycopg.errors.UniqueViolation`
- constraint: `processed_message_ledger_service_name_source_topic_source_p_key`

Crash loop prevented end-to-end ingestion continuity and blocked multi-instrument flow from appearing in downstream tables.

## Impact

- `market-normalization` entered restart loop.
- `raw_tick` and `ohlcv_bar` updates stalled or became inconsistent.
- Operator observed only partial symbol ingestion despite multi-symbol `.env` configuration.

## Reproduction

1. Run ingestion stack with `market-ingestion` and `market-normalization`.
2. Ensure at least one offset for the same `(service_name, source_topic, source_partition, source_offset)` already exists in `processed_message_ledger`.
3. Re-consume or reprocess the same message offset.
4. Observe `market-normalization` crash with duplicate key violation.

## Root Cause Analysis

`record_message_processed()` inserted into `processed_message_ledger` with:

- `ON CONFLICT (service_name, message_id) DO NOTHING`

But table uniqueness also includes offset identity:

- `(service_name, source_topic, source_partition, source_offset)` unique constraint

When duplicate offset arrived with different `message_id`, conflict occurred on the offset constraint and was not handled by the narrower conflict target.

## Resolution Implemented

Code fix:

- File: `services/ingestion/contracts.py`
- Change: switched to `ON CONFLICT DO NOTHING` for `processed_message_ledger` insert.

This preserves idempotency for both message-id and offset-based duplicate paths without crashing consumer services.

## Verification Evidence

- `market-normalization` traceback previously showed repeated duplicate-key failures on offset-unique constraint.
- After fix and service restart, insert duplicates are ignored instead of raising an exception.
- Service remains up and continues processing records.

## Notes

- This fix is shared by all services using `record_message_processed()`.
- Runtime recovery requires restarting impacted ingestion services after deploying the code change.
